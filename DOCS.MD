## 6. Key Functionality Deep Dive / User Flows

### 6.1. Starting the Bot & Main Menu
1.  User sends `/start` or interacts with the bot for the first time.
2.  `start()` handler:
    *   Clears any existing `context.user_data` for a clean session.
    *   Sets `context.user_data['session_started'] = True` to indicate an active session.
    *   Sends a welcome message mentioning the user.
    *   Displays the `main_menu_markup` (ReplyKeyboardMarkup) with options:
        *   "📚 Выбрать тему и начать"
        *   "📊 Мой прогресс"
        *   "❓ Помощь"
3.  If a user sends any text message (not a known command like `/stop`) *before* `/start` has been used in the current session (i.e., `session_started` is not `True`):
    *   `handle_message()` will intercept this.
    *   The bot will reply with "Привет! Чтобы начать, пожалуйста, отправь команду /start."
    *   The main menu keyboard will *not* be shown at this point.
4.  If a user sends any text message (not a known command or a main menu option) *after* `/start` has been used:
    *   And no test is active: `handle_message()` replies with "Выбери действие из меню:" and the main menu keyboard.
    *   And a test is active: `handle_message()` replies with guidance to use test buttons.

### 6.2. Selecting a Topic & Starting a Test
1.  User clicks "📚 Выбрать тему и начать" from the main menu (assumes `/start` was already used).
2.  `handle_message()` detects this text.
    *   Checks if user is active using `db.is_user_active(user_id)`
    *   If active, prevents starting a new test and guides user to complete current test
    *   If not active, proceeds with topic selection
3.  `show_topics_from_message()`:
    *   Attempts to remove the `ReplyKeyboardMarkup`.
    *   Sends a message "📚 Выбери тему:" with an `InlineKeyboardMarkup` generated by `_build_topic_selection_keyboard()`. This lists all topics from `constants.TOPICS` and a "🏠 В главное меню" button.
4.  User clicks on a topic button (e.g., `callback_data="topic_0"`).
5.  `start_test()` handler:
    *   Answers the callback query.
    *   Retrieves the selected topic.
    *   **Shows a loading message immediately:** Edits the message to "⏳ Загрузка вопросов...".
    *   **Starts question generation in the background** (does not block other users).
    *   When questions are ready, starts the test for the user.
    *   If no questions could be loaded, shows an error message and deactivates the user.

### 6.3. Answering Questions
1.  `ask_question()`:
    *   Checks if the current question index is within bounds. If not, calls `show_results()`.
    *   If viewing an *already answered question* (e.g., after navigating with "Предыдущий"/"Следующий"):
        *   Displays the previous feedback.
        *   Always includes navigation buttons ("⬅️ Предыдущий", "➡️ Следующий" if applicable) and "⬅️ Назад к темам".
    *   If the question is *new/unanswered*:
        *   Retrieves question data.
        *   Prepares and shuffles answer options.
        *   Constructs an `InlineKeyboardMarkup` with answer buttons and navigation buttons ("⬅️ Предыдущий", "➡️ Следующий" if applicable).
        *   The "⬅️ Назад к темам" button is added *only if it's the first question of the test* (`q_num == 0`). For subsequent unanswered questions (Q2, Q3, etc.), this button is omitted.
        *   Displays the question and options.
2.  User clicks an answer button.
3.  `check_answer()` handler:
    *   Answers the callback query.
    *   Parses `callback_data` to get selected option index and question number.
    *   Ignores if it's an outdated answer or already processed.
    *   Retrieves the selected option text from the button itself.
    *   Compares user's answer with the correct answer (case-insensitive, trimmed).
    *   Updates `context.user_data['correct_answers']`.
    *   If incorrect:
        *   Logs the error details in `context.user_data['errors']`.
        *   Saves the error to the `user_question_errors` table in the database using `db.add_user_error()`.
    *   Stores the fact that this question has been answered and the user's choice in `context.user_data['answered_questions']`.
    *   Constructs feedback text (correct/incorrect, correct answer, explanation preview if short).
    *   Displays feedback with a "➡️ Продолжить" button.
4.  User clicks "➡️ Продолжить".
5.  `continue_test()` handler:
    *   Answers the callback query.
    *   Increments `context.user_data['current_question']`.
    *   Updates `context.user_data['max_reached_question']`.
    *   Calls `ask_question()` for the next question or `show_results()` if the test is complete.

### 6.4. Viewing Test Results & Explanations
1.  `show_results()` (called after the last question or if test ends prematurely):
    *   Answers callback query if any.
    *   Calculates score and percentage.
    *   Saves the test result (percentage) to `test_history` table using `db.add_test_result()`.
    *   Constructs a results message.
    *   If errors occurred:
        *   Lists each error with a button "Показать объяснение к вопросу X" (`callback_data="show_expl_X"`).
    *   Provides buttons: "🏠 В главное меню", "🔄 Пройти еще раз эту тему", "📚 Выбрать другую тему".
    *   Displays the results.
2.  User clicks "Показать объяснение к вопросу X".
3.  `show_explanation()` handler:
    *   Retrieves the specific error details (question, explanation) from `context.user_data['errors']`.
    *   Constructs the full explanation text.
    *   If a previous explanation message ID is stored (`last_explanation_message_info_on_results`), attempts to edit it.
    *   Otherwise, sends a new message replying to the results message and stores its ID and chat ID.
    *   Navigating away from the results screen (e.g., to the main menu or back to topic selection) will also delete any currently displayed explanation message.

### 6.5. Checking Progress
1.  User clicks "📊 Мой прогресс" from the main menu (assumes `/start` was already used).
2.  `handle_message()` detects this.
    *   Note: `context.user_data.clear()` is no longer called here.
3.  `show_progress_from_message()`:
    *   Fetches overall progress (`COUNT(*)` tests, `AVG(result)`) from `test_history`.
    *   Fetches sum of `error_count` from `errors` table.
    *   Fetches recent topics (`topic, result, date`) from `test_history`.
    *   Formats and sends this information to the user with the main menu `ReplyKeyboardMarkup`.

### 6.6. Getting Help
1.  User clicks "❓ Помощь" from the main menu.
2.  `handle_message()` detects this.
3.  The `handle_message` function directly replies with the comprehensive `HELP_TEXT` from `constants.py` and the `main_menu_markup`.
    The `HELP_TEXT` now provides detailed information on:
    *   Starting a test and selecting topics.
    *   Answering questions and receiving feedback/explanations.
    *   Navigating within a test (previous/next question, returning to topics from the first question).
    *   Viewing test results and detailed error explanations.
    *   Checking overall progress.
    *   Returning to the main menu from different screens.
    *   Using `/start` to refresh the session.
    *   Using `/stop` to delete all personal data.
    *   A brief note on question sourcing (database and AI).

### 6.7. Returning to Main Menu from Inline Keyboards
1.  User clicks a button with `callback_data="main_menu"` (e.g., from results screen or topic selection).
2.  `go_to_main_menu_callback()`:
    *   Answers the callback query.
    *   Attempts to remove the inline keyboard from the message it originated from.
    *   Deletes any active explanation message if `last_explanation_message_info_on_results` is present in `context.user_data`.
    *   Sends a new message "✅ {user.mention_html()}, вы снова в главном меню..." with the `main_menu_markup`.

### 6.8. Stopping the Bot & Deleting User Data (New)
1.  User sends the `/stop` command.
2.  `stop_command()` handler:
    *   Retrieves the `user_id` and `chat_id`.
    *   **Checks if a session was started**: It verifies `context.user_data.get('session_started')`.
        *   **If session NOT started**: 
            *   Logs this event.
            *   Sends a message: "Бот еще не был активирован командой /start. Для вас нет сохраненных данных. Чтобы начать, отправьте /start."
            *   Returns, without attempting data deletion.
        *   **If session WAS started**:
            *   Logs that `/stop` was executed for an active session.
            *   Calls `db.set_user_inactive(user_id)` and `db.delete_all_user_data(user_id)` to remove all data associated with the user from the database, including active sessions.
            *   Clears any session data from `context.user_data`.
            *   Sends a confirmation message to the user indicating that their data has been deleted (this message also includes `ReplyKeyboardRemove`).
            *   Informs the user that the bot cannot delete the chat history itself.
            *   Suggests sending `/start` to begin interacting with the bot again.

### 6.9. Returning to Topic Selection from Results/Test
1.  User clicks a button with `callback_data="back_to_topics"`:
    *   This button appears on the results screen, on any *answered* question screen (if navigated back to), and on the *first unanswered* question screen.
    *   If clicked, `back_to_topics()` handler is called:
        *   Answers the callback query.
        *   If called from the results screen where an explanation might be visible, it attempts to delete the explanation message.
        *   **Clears all `context.user_data` and marks the user as inactive in the database.**
        *   Calls `show_topics()` to display the topic selection.
2.  If a user attempts to use Main Menu commands (like "📚 Выбрать тему и начать") during an active test:
    *   `handle_message()` informs them: "Вы проходите тест. Чтобы выбрать другую опцию, пожалуйста, завершите текущий тест. Для возврата к выбору тем без завершения теста, перейдите к первому вопросу теста." This guides them to use "⬅️ Предыдущий" to return to the first question if they wish to exit early via the "⬅️ Назад к темам" button available there.

## 7. Database Schema (`database.py`)

*   **`users`**:
    *   `user_id` (INTEGER, PRIMARY KEY): Telegram User ID.
    *   `name` (TEXT): User's name (currently not actively populated).
    *   `grade` (INTEGER): User's grade level (currently not actively populated).

*   **`active_users`**:
    *   `user_id` (INTEGER, PRIMARY KEY): Telegram User ID.
    *   `current_topic` (TEXT): Current test topic.
    *   `current_question` (INTEGER): Current question number.
    *   `start_time` (TIMESTAMP): When the test was started.
    *   `last_activity` (TIMESTAMP): Last user activity.
    *   FOREIGN KEY to `users (user_id)`.

*   **`test_history`**:
    *   `id` (INTEGER, PRIMARY KEY AUTOINCREMENT)
    *   `user_id` (INTEGER, FOREIGN KEY to `users.user_id`)
    *   `topic` (TEXT)
    *   `date` (TIMESTAMP)
    *   `result` (INTEGER): Percentage score.

*   **`errors`** (Summary of errors per topic):
    *   `id` (INTEGER, PRIMARY KEY AUTOINCREMENT)
    *   `user_id` (INTEGER, FOREIGN KEY to `users.user_id`)
    *   `topic` (TEXT)
    *   `error_count` (INTEGER): Number of times questions in this topic were answered incorrectly.
    *   `UNIQUE(user_id, topic)`

*   **`user_question_errors`** (Detailed log of specific errors):
    *   `id` (INTEGER, PRIMARY KEY AUTOINCREMENT)
    *   `user_id` (INTEGER, FOREIGN KEY to `users.user_id`)
    *   `topic` (TEXT)
    *   `question_text` (TEXT)
    *   `user_answer_text` (TEXT)
    *   `correct_answer_text` (TEXT)
    *   `explanation_text` (TEXT)
    *   `timestamp` (TIMESTAMP)

*   **`tasks`** (Question bank):
    *   `id` (INTEGER, PRIMARY KEY AUTOINCREMENT)
    *   `question` (TEXT)
    *   `answer` (TEXT): Correct answer.
    *   `explanation` (TEXT)
    *   `topic` (TEXT)
    *   `level` (INTEGER): Difficulty level (currently not actively used for filtering).
    *   `incorrect_options` (TEXT, DEFAULT NULL): JSON string list of AI-generated incorrect options.

## 8. User Session Management

The bot now implements sequential user processing through the following mechanisms:

1. **Active User Tracking**:
   * Users are marked as active when they start a test
   * Their activity status is tracked in the `active_users` table
   * **Each user can only have one active test at a time, but all users are independent**

2. **Session Control**:
   * Users cannot start a new test while another is in progress **(for themselves only)**
   * Session timeout is tracked through `last_activity` timestamp
   * Users are automatically marked as inactive when they complete a test or send `/stop`

3. **Error Management**:
   * Each user's errors are stored в their own table (`user_question_errors`)
   * Error statistics are maintained per topic in the `errors` table
   * The system prioritizes showing questions that users previously answered incorrectly

4. **Question Bank**:
   * All questions are stored in a common `tasks` table
   * Questions can be generated by AI or manually added
   * **Questions are shared across all users but error tracking and test progress are individual**

## 9. AI Integration (Google Gemini)

*   **Model Initialization:** `genai.GenerativeModel(os.getenv('GEMINI_MODEL'))`
*   **Task Generation (`generate_ai_task`)**:
    *   **Parallel Generation:** Questions are generated in parallel using `asyncio.gather()` for better performance
    *   **True Async:** AI question generation is now fully asynchronous using `run_in_executor`, so multiple users can start and pass tests independently, even if AI is slow.
    *   **Retry Logic:** The bot will make several attempts to generate the required number of unique, valid questions, so the test will almost always be full-length unless the AI is completely broken.
    *   **Prompt Engineering:** A detailed prompt is used, instructing the AI to generate:
        1.  Question text (for 5-6th grade), emphasizing variety (word problems, logic puzzles, conceptual applications) rather than simple calculations or number swaps.
        2.  One correct answer (clear, unambiguous, <= 60 chars).
        3.  Three plausible incorrect answer options, reflecting common student errors.
        4.  A brief, clear, and potentially step-by-step explanation.
        *   Specific constraints are given (e.g., answer unit consistency, appropriate difficulty solvable mentally or with minimal writing, novelty).
        *   **Strict Formatting Requirement:** AI is asked to format each part with labels like "ВОПРОС:", "ПРАВИЛЬНЫЙ ОТВЕТ:", etc., each on a new line, with ALL CAPS headers. An example of the desired structure and question type is provided to the AI.
    *   **Response Parsing:**
        *   Regular expressions (`re.search`, `re.finditer`) are used to extract the question, correct answer, incorrect options, and explanation based on the predefined labels.
        *   Error handling is in place if parsing fails or critical parts are missing.
    *   **Output:** Returns `(question, correct_answer, incorrect_options_list, explanation)`.
*   **Usage:**
    *   Called by `get_or_generate_tasks` when new unique questions are needed.
    *   Successfully generated tasks are stored in the `tasks` database table, including the `incorrect_options` as a JSON string.
    *   Questions are generated in parallel for better performance.

## 11. Known Limitations / Areas for Attention

*   **AI Reliability:** AI-generated content might occasionally be inaccurate, poorly phrased, or not perfectly match the requested format, despite prompt engineering.
*   **Parsing Fragility:** Relies on strict AI output format for parsing. Changes in AI behavior could break this.
*   **Scalability of SQLite:** For very high concurrent user counts, SQLite might become a bottleneck.
*   **Content Management:** Adding new topics or manually vetting AI questions currently requires direct code/DB modification.
*   **"Universal Options":** The `generate_universal_options` function's numeric distractor generation logic is quite complex and might sometimes produce non-ideal options. The generic text fallbacks are very basic.
*   **`MAX_EXPLANATION_PREVIEW_LENGTH`**: If an explanation is slightly longer than this, the user only sees "Подробное объяснение будет доступно на экране результатов," which might be frustrating for an immediate small overflow.
*   **Hardcoded Values**: `DEFAULT_QUESTIONS_PER_TEST` is hardcoded. User configuration could be a feature.
